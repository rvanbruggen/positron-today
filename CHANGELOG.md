# Changelog

All notable changes to Positron Today. Newest first.

| Version | Highlights |
|---------|-----------|
| **2.22.4** | Fix pipeline resume on tab reopen. The mount check was calling `GET /api/pipeline/status`, which is NOT in the proxy's public paths — so the request was likely being rejected (401/redirect), with the error swallowed by the catch block. Switched to `POST /api/pipeline/tick` for the mount check, which IS whitelisted in public paths and also processes a step immediately on resume. The tick endpoint auto-discovers the active run when called without a runId, returns full status including logs, and handles all three states (running → resume polling, error/done → show final state) |
| **2.22.3** | Show the last pipeline run's final state on mount even when it's already finished. Previously the Preview page's on-mount check only attached to runs with `status='running'` — if the pipeline errored or completed while the tab was closed, reopening showed nothing (no logs, no error, no progress). Now the mount check also displays recently finished runs (error or done, within the last 10 minutes): fetches the full run including logs, renders the log panel and any error messages, so the user always sees what happened. Running pipelines still resume ticking as before |
| **2.22.2** | Switch to browser-driven pipeline with cron backup for background execution. The `after()` + self-call chain from 2.22.1 bypassed Vercel's Deployment Protection (via the automation bypass header) but then hit Vercel's **infinite loop detection** (508) — the platform tracks chained requests and kills them after ~5 hops, making recursive self-calls unviable regardless of auth. Final architecture: the browser drives the pipeline via `POST /api/pipeline/tick` (each poll processes one step and returns updated status), with an in-flight guard preventing overlapping ticks. For background execution (mobile tab suspended), `/api/pipeline/tick` is added to the proxy's public paths so an external cron (Synology NAS) can call it periodically to keep the pipeline moving. The tick endpoint auto-discovers the active run when called without a `runId`, so the cron needs no parameters. The `after()` + `selfFetch()` mechanism is fully removed from both start and step routes. Trade-off vs. the original 2.20.0 design: background progress depends on the cron interval (e.g. every 1–2 minutes) rather than chaining immediately, but it's reliable across all Vercel constraints |
| **2.22.1** | Restore the server-side `after()` pipeline chain and fix the Vercel 401 using the Protection Bypass for Automation header. The root cause of the persistent 401 across 2.21.1–2.21.6 was Vercel's Standard Deployment Protection — an infrastructure-level auth layer that rejects requests before application code runs, making proxy whitelists, Cookie headers, and Authorization headers ineffective. Fix: `selfFetch()` now includes the `x-vercel-protection-bypass` header using the `VERCEL_AUTOMATION_BYPASS_SECRET` environment variable, which tells Vercel's edge to let the request through to the application. The `after()` + self-call chain is restored so the pipeline runs entirely server-side and survives mobile backgrounding. The browser-driven tick approach from 2.22.0 is kept as a fallback endpoint but no longer used by default. Also improved error diagnostics: self-call failures now include the first 200 chars of the response body in the error message |
| **2.22.0** | Eliminate server-to-server HTTP self-calls from the pipeline, fixing the persistent 401 auth failure on Vercel. The `after()` + `selfFetch()` architecture (introduced in 2.20.0) made each pipeline step call the next via an HTTP POST to the same deployment — but Vercel's infrastructure-level authentication (Deployment Protection or edge-layer auth) rejected these server-to-server requests before they reached the application's own proxy code, producing an unbypassable 401 regardless of Cookie headers, Authorization headers, proxy whitelists, or matcher exclusions (six failed attempts across 2.21.1–2.21.6). New architecture: the browser drives the pipeline. Each 2-second poll now calls `POST /api/pipeline/tick`, which processes one step (one RSS chunk, one classify batch, or the export) and returns the updated status — no server-to-server calls, no `after()`, no auth issues. An in-flight guard on the client prevents overlapping ticks. Step processing logic extracted into shared `admin/lib/pipeline-steps.ts` module; the standalone `POST /api/pipeline/step` endpoint is retained for debugging but no longer used by the pipeline chain. Trade-off: the pipeline pauses when the browser tab is closed (no background server chain), but resumes automatically when the user returns to the Preview page — the on-mount check detects the running pipeline and restarts polling. The stale-run 10-minute auto-stop remains as a safety net |
| **2.21.6** | Fix pipeline self-call auth on Vercel by switching from `Cookie` to `Authorization: Bearer` header. Vercel's edge/serverless runtime strips `Cookie` headers from outgoing `fetch()` requests made within `after()` callbacks, so the HMAC session token added in 2.21.5 never reached the proxy. The proxy in `proxy.ts` now accepts `Authorization: Bearer <token>` as an alternative to the session cookie, and `selfFetch()` in `admin/lib/self-url.ts` sends the token via `Authorization` instead of `Cookie`. The token computation is identical — HMAC-SHA256 of `ADMIN_SECRET` over the fixed string `"positron-admin-v1"` — only the transport header changed |
| **2.21.5** | Fix pipeline auth by including the session cookie in self-calls. Neither the proxy function's `publicPaths` whitelist (2.21.1) nor the matcher regex exclusion (2.21.4) prevented the 401 on Vercel — changes to `proxy.ts` appear not to reach the deployed proxy function. New approach: `selfFetch()` helper in `admin/lib/self-url.ts` computes the same HMAC-SHA256 session token the proxy expects (using `ADMIN_SECRET` from the environment) and includes it as a `Cookie` header on every pipeline self-call. The proxy sees a valid session and lets the request through, regardless of its whitelist or matcher configuration. Reverted the matcher change from 2.21.4 |
| **2.21.4** | Restore HTTP self-calls for the pipeline chain (reverts 2.21.3's direct-call approach which ran the entire pipeline in a single Vercel function invocation, easily exceeding the 60s `maxDuration`). Fix the auth issue at the **matcher** level instead: exclude `api/pipeline/step` from the proxy's regex matcher in `proxy.ts` so the proxy function never even runs for self-calls. The in-function `publicPaths` whitelist added in 2.21.1 apparently didn't take effect (possibly a Vercel build-cache issue with the proxy function); the matcher exclusion operates at the Next.js routing layer before the proxy code is invoked, making it more reliable. Each pipeline step is now a separate Vercel function invocation again, each with its own 60s budget — safe for 41+ sources |
| **2.21.3** | Eliminate HTTP self-calls in the pipeline chain. The `after()` callbacks in start and step routes were calling `POST /api/pipeline/step` via HTTP — but the proxy (auth middleware) blocks cookieless requests, and adding the path to the public whitelist didn't reliably take effect on Vercel. Root-cause fix: extract a `processStep()` function from the step route and call it directly from `after()` — no HTTP round-trip, no proxy, no auth. The `chainNext()` helper and the start route's `after()` both now invoke `processStep()` in-process. The HTTP endpoint (`POST /api/pipeline/step`) still exists for debugging but is no longer used by the chain. Reverted the proxy.ts whitelist change from 2.21.1 since it's no longer needed |
| **2.21.2** | Surface pipeline errors in the UI. When a run enters `status='error'` before any log entries are written (e.g. the `after()` self-call fails), the poll now injects the `error_message` as a visible fatal log line so the panel stays on screen instead of silently vanishing. Both `after()` catch blocks (in start and step routes) now also write the error to the `log` column so it's available for the poll |
| **2.21.1** | Fix pipeline self-call being blocked by auth proxy on Vercel. The `after()` callbacks in `/api/pipeline/start` and `/api/pipeline/step` make server-to-server `fetch()` calls to `/api/pipeline/step` to chain the next processing step — but these calls carry no session cookie. The auth proxy in `proxy.ts` wasn't whitelisting `/api/pipeline/step` as a public path (unlike the older `/api/fetch-feeds` and `/api/classify` endpoints), so on Vercel the self-call received a 401 Unauthorized response. Because `fetch()` only rejects on network errors (not HTTP errors), the catch block never fired, the chain died silently, and the run stayed in `status='running'` with `phase='fetch'` and `sources_done=0` forever — producing the permanent "Phase 1 — scanning source 1 of N" display. Fix: add `/api/pipeline/step` to the proxy's public paths. Also harden both `after()` callbacks to check `res.ok` and throw on non-2xx responses, so future auth or routing issues surface as proper error states instead of silent zombies |
| **2.21.0** | Pipeline stop button and stale-run recovery. **(1) Stop endpoint + UI button.** New `POST /api/pipeline/stop` sets the active run's status to `'error'` with message "Cancelled by user", appends a `fatal` log entry so the UI renders it, and drains leftover `pending_items`. The existing step route already checks `status === 'running'` before proceeding, so the chain stops naturally on the next tick. The Preview page now shows a red **Stop** button next to "Fetching…" while a pipeline is running, and the progress label switches to "Stopped by user" on cancellation. **(2) Stale-run auto-recovery.** Two layers protect against permanently stuck pipelines (e.g. the `after()` chain silently dying on Vercel). The `/api/pipeline/start` endpoint now checks the age of any existing "running" row — if older than 10 minutes, it force-stops it as "Timed out (stale run)" and proceeds to create a fresh run instead of returning 409. The Preview page's on-mount check does the same: if it finds a running pipeline older than 10 minutes, it auto-stops it via the stop endpoint rather than re-attaching to a zombie. Together these ensure the UI never gets permanently stuck showing Phase 1 of a dead run |
| **2.20.1** | Fix the server-side pipeline's progress bar never updating after the initial "source 1 of N" state. Three root causes. **(1) Missing `maxDuration` in `vercel.json`.** The new `pipeline/start` and `pipeline/step` routes added in 2.20.0 were not listed in `vercel.json`, so they ran with Vercel's default function timeout (~10-15s) instead of the intended 60s — easily exceeded when processing 15 RSS feeds with per-item duplicate checks against Turso, causing the function to be killed before it could update counters or chain the next step. Added `maxDuration: 60` for both routes. **(2) Unquoted `offset` column in dynamic SQL.** The `updateRun()` helper in `pipeline/step` built SET clauses with bare column names (`offset = ?`); `OFFSET` is a SQL keyword and could cause the UPDATE to error on some SQLite/libsql versions, silently preventing `sources_done` and other counters from being persisted. All column names in `updateRun()` are now double-quoted (`"offset" = ?`). The `pipeline/status` SELECT query had the same issue and is also quoted. **(3) Stale polling responses.** The `pipeline/status` GET endpoint had no caching directive, so Next.js production builds and/or browser caches could serve stale progress data. Added `export const dynamic = "force-dynamic"` to the route and a `&t=` cache-buster to the polling fetch in the Preview page UI |
| **2.20.0** | Move the fetch-and-classify pipeline orchestration from client-side to server-side so closing the browser or backgrounding the mobile app no longer interrupts a run. Previously the Preview page's "Fetch new articles" button drove a multi-request loop from JavaScript — calling `/api/fetch-feeds` in chunks then `/api/classify` in batches, waiting for each response before firing the next. On desktop browsers that worked fine (the tab keeps running), but on mobile the OS suspends background tabs aggressively, killing the orchestration mid-flight. Now the button fires a single POST to a new **`/api/pipeline/start`** endpoint, which creates a `pipeline_runs` row in the DB and returns a `runId` immediately. The actual work happens server-side via **`/api/pipeline/step`**, a self-chaining endpoint that uses Next.js `after()` to fire the next step after each response completes — fetch chunk 1 → fetch chunk 2 → … → classify batch 1 → classify batch 2 → … → export rejection log → mark done. Each step fits comfortably within Vercel's 60s Hobby-tier `maxDuration`; `after()` ensures the next invocation is triggered even after the HTTP response is sent and the client has disconnected. The UI polls **`/api/pipeline/status?runId=X`** every 2 seconds for progress, log lines, and phase information; the progress bar and NDJSON-style log panel render identically to before. If the user navigates away and returns to the Preview page, an on-mount check detects any running pipeline and resumes polling automatically — no work is lost. New `pipeline_runs` table (idempotent migration in `admin/lib/schema.ts`) tracks status, phase, cumulative counters (queued/classified/added/filtered/errored), queue depth, and a JSON log column. New `admin/lib/self-url.ts` helper resolves the app's own base URL for server-to-server calls (`VERCEL_URL` on Vercel, `localhost` in dev). Overlap protection: `/api/pipeline/start` returns 409 if a run is already in progress, and the UI attaches to the existing run instead of starting a duplicate. All existing endpoints (`/api/fetch-feeds`, `/api/classify`, `/api/fetch`) are untouched — the Synology cron and any other external callers continue to work exactly as before |
| **2.19.0** | Split the RSS fetch pipeline into two phases so a growing source list (now 42, with more arriving in non-EN/NL/FR languages) stops bumping into Vercel's 60s `maxDuration` ceiling. The legacy `/api/fetch` route did everything in one invocation — pull RSS → dedup → LLM positivity filter — and the LLM call dominates per-item cost, so a single chunk could blow the budget if a few sources happened to have many genuinely new items at once (each classify ≈ 1–3s, easily 50 calls per chunk in the worst case). Now there are three logical steps as two HTTP endpoints. **(1) `/api/fetch-feeds` — Phase 1, no LLM.** Pulls RSS for a chunk of sources (default `?limit=15`, larger than the legacy `5` because no LLM means each source costs ~1–2s worst-case bounded by the existing 8s `RSSParser` timeout), dedups every item against `raw_articles` / `rejected_articles` / `articles` AND a new `pending_items` queue table, then inserts genuinely new items into `pending_items`. Streams the same NDJSON shape as `/api/fetch` so the UI can show progress in real time. Auto-mode rotates through chunks via the existing `auto_fetch_offset` setting, identical to before. **(2) `/api/classify` — Phase 2, LLM-bounded.** Pulls a batch from `pending_items` (default `?limit=15`, JOINed against `sources` for the language column so per-item DB chatter is one query not two), runs each through the positivity filter, and routes the result into `raw_articles` (accepted) or `rejected_articles` (rejected) — same logic, prompt, and translation handling as the legacy route — then deletes the queue row. Errors leave the row in place so a future run retries naturally. The rejection-log export to the public site (`exportRejections()`) moved here, fired only when the queue fully drains, so it runs once per cycle instead of per chunk. New `pending_items` table is a 6-column staging area with `UNIQUE(url)` and an `ON DELETE CASCADE` to `sources`; idempotent migration in `admin/lib/schema.ts`. **(3) Preview page button drives both phases sequentially.** The "Fetch new articles" button on `/preview` now loops `/api/fetch-feeds` chunks until all sources are exhausted, then loops `/api/classify` batches until the queue is empty. The progress bar splits 50/50 between the two phases and the NDJSON log stream interleaves Phase 1 and Phase 2 events with a "▸ Phase N" sectioning header. New summary bar at the bottom aggregates `added` / `filtered` / `queued` / `skipped` / `errored` across every chunk and batch — previously only showed the last `done` event, which lost information when chunking was active. **(4) Legacy `/api/fetch` kept operational** for the Synology cron's transition window — its dedup check now also looks at `pending_items` so the two pipelines can't fight over the same URL. Recommended migration: replace any `/api/fetch?auto=1` cron with two entries (`/api/fetch-feeds?auto=1` and `/api/classify?auto=1`); they can run on the same schedule or different ones, and items survive in `pending_items` between calls so neither needs to "complete" inside any single 60s window. New routes whitelisted in `proxy.ts` and added to `vercel.json` with `maxDuration: 60` |

| **2.16.5** | Stop the `/api/fetch` route from getting stuck against Vercel's 60s `maxDuration` cap on the Hobby plan. Vercel logs were showing recurring `FUNCTION_INVOCATION_TIMEOUT` (504) on `app/api/fetch/route.ts`, with the streamed NDJSON response cut off mid-chunk and the auto-fetch offset advancing past sources that hadn't actually been processed. Three reliability fixes in `admin/app/api/fetch/route.ts`. **(1) Per-feed timeout on `RSSParser`.** `rss-parser` ships with no default timeout, so a single hanging RSS host (slow DNS, dead origin holding the TCP connection open, etc.) could eat the entire 60s budget while the other 9 sources in the chunk never got a chance to run. Net effect: worst-case per-invocation time is now ≈ 5 × max(8s feed timeout, classify cost for ≤10 items) — well clear of 60s on a Hobby plan |

| **2.16.2** | Defensive fix in `OpenAIProvider.call()` (`admin/lib/llm.ts`) so the positivity filter and summariser don't break if the user selects an OpenAI reasoning model (`o1*`, `o3*`, …) for either task on the Settings page. | **2.16.1** | Make the positivity-filter `classify()` path deterministic across all three providers. The Anthropic implementation in `admin/lib/llm.ts` previously sent no `temperature` parameter at all, so the API default (~1.0) applied — meaning the same article + prompt could classify YES one minute and NO the next, and rules buried mid-paragraph in the filter prompt (sports rejection, "global relevance" preference) were getting skipped on borderline cases instead of consistently enforced. The Ollama and OpenAI implementations both passed `0.3`, also more than ideal for a binary classification call. All three `classify()` paths now pass `temperature: 0` for a stable yes/no decision.
